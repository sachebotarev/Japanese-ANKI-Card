# Правила для списков слов

Эти правила применяются к файлам в каталоге `Слова/` до любой дальнейшей обработки.

## Нормализация

- Удалить пустые строки и строки, состоящие только из пробелов.
- Удалить точные дубли строк с сохранением регистра и символов.
- Отсортировать строки в порядке Unicode, лексикографически по возрастанию.
- Сохранить файл так, чтобы в нём оставалась ровно одна запись на строку.

## Связь со структурой проекта

- Каждый файл в `Слова/` задаёт тему.
- Имя темы должно совпадать с поддиректорией в `Карточки/<тема>/`.
- То же имя темы используется в `Картинки/<тема>/` и `Произношение/<тема>/`.
- Только после нормализации список слов используется для создания карточек.

## Выбор темы для нового слова

- Если пользователь указывает конкретный файл или тему, слово нужно класть туда, если это не противоречит явной структуре проекта.
- Если пользователь даёт слово без указания темы, тему нужно определить самостоятельно по смыслу слова.
- Если слово является существительным, его нужно по возможности классифицировать по теме применения, а не складывать автоматически в тему `Существительные`.
- Для существительных сначала нужно искать уже существующую тематическую категорию. Например, слова о частях тела нужно относить в `Части тела`, слова о профессиях — в `Профессии`, а не в общий файл `Существительные`.
- Тему `Существительные` нужно использовать только как запасной вариант для существительных, которым пока не находится более точная и устойчивая тема среди уже существующих.
- После выбора темы слово нужно реально записать в соответствующий файл `Слова/<тема>`, а не только использовать эту тему для создания JSON-карточки, если для данного файла нет отдельного запрета.
- Сначала нужно пытаться использовать наиболее подходящую уже существующую тему.
- Новую тему и новый файл в `Слова/` нужно создавать только тогда, когда среди существующих тем нет разумно подходящей.
- Для существительных новую тему нужно создавать только если она описывает внятную область применения и помогает собрать не единичное слово, а компактную полезную группу.
- Не нужно плодить отдельные темы ради одного слова, если его можно без натяжки отнести к уже существующей категории.
- При выборе между идеальной точностью и разрастанием структуры нужно придерживаться разумного компромисса в пользу устойчивой и компактной организации проекта.

## Короткие пользовательские запросы

- Короткий запрос вида `Добавь слово ...` или `Создай карточку для слова ...` означает, что тему для слова нужно определить самостоятельно по правилам этого файла.
- Для такого короткого запроса недостаточно просто выбрать тему мысленно: если тема не подпадает под отдельное исключение, агент должен обновить сам файл `Слова/<тема>`, затем нормализовать его и только после этого продолжать полный пайплайн.
- Если пользователь перечисляет несколько слов в одном коротком запросе, каждое слово нужно классифицировать отдельно, но по возможности не дробить структуру без необходимости.

## Особое правило для `Слова/Кандзи`

- Файл `Слова/Кандзи` не пополняется автоматически агентом.
- Строки в `Слова/Кандзи` добавляет только пользователь вручную.
- Агент не должен дописывать туда слова даже в тех случаях, когда по смыслу тема `Кандзи` кажется подходящей, если пользователь отдельно не попросил об этом.
- Прямой просьбой для этого исключения считается только явное указание добавить слово именно в `Слова/Кандзи` или явно разрешить изменение этого файла. Формулировки вроде `в нужный файл`, `добавь слово` или `сделай полный пайплайн` не снимают запрет.
- Это исключение не отменяет общего правила выбора темы: агент всё равно должен выбрать тему `Кандзи`, создать карточку в `Карточки/Кандзи/` и явно сказать пользователю, что шаг с обновлением `Слова/Кандзи` пропущен именно из-за ручного режима этого файла.
- При этом карточки и сопутствующие файлы по теме `Кандзи` создавать можно, если это не противоречит другим явным указаниям пользователя.
